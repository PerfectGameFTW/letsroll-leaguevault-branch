# ESLint baseline ceiling bump log

This file is the public-facing audit log for every upward bump of the
ESLint suppression ceilings enforced by
[`scripts/check-eslint-baseline.ts`](scripts/check-eslint-baseline.ts).

The ceilings themselves live as literals at the top of that script, and
the live suppression counts are tracked in
[`eslint-suppressions.json`](eslint-suppressions.json) (regenerated by
ESLint's suppression command). The script blocks any commit that sets a
`RULE_CEILINGS[<rule>]` (or `TOTAL_CEILING`) **above** the most recent
`new ceiling` recorded in the table below for that rule.

## How to bump a ceiling

1. Decide whether the bump is genuinely necessary. The default is to
   remove the new suppression(s) by typing the offending code instead.
2. If the bump must happen, add a new row to the table below in the
   same commit. The row must include:
   - `rule` — the rule key as it appears in `RULE_CEILINGS` (use
     `TOTAL` for the aggregate ceiling).
   - `old ceiling` — the previous recorded value (last `new ceiling`
     for that rule in this table).
   - `new ceiling` — the value you are setting in the script.
   - `delta` — `new - old` (positive for a bump, signed).
   - `reason` — one short sentence explaining why the bump is
     justified (e.g. "test mocks for new admin email-change UI").
   - `commit/task ref` — the task number (e.g. `#684`) or commit SHA
     that introduced the suppressions.
3. Update `RULE_CEILINGS` / `TOTAL_CEILING` in
   `scripts/check-eslint-baseline.ts` to match the new value.
4. Run `tsx scripts/check-eslint-baseline.ts --strict` to confirm the
   gate accepts the change.

Decreases (ratchet-downs) and unchanged ceilings do **not** require a
new row — the table only grows on bumps. The script reads the most
recent row per rule as the "recorded ceiling"; as long as the script's
literal is `<=` the recorded value, the gate stays green.
